# Backend Developer — подробная роль (4everCare)

Документ для студента, отвечающего за backend: что делал, где код, что говорить на защите.

---

## 1. Зона ответственности

- **Настройка приложения**: создание Flask-приложения, конфигурация, подключение БД.
- **Маршруты (routes)**: все URL и обработка GET/POST.
- **Бизнес-логика**: валидация формы, правила бронирования и отмены, защита от двойного бронирования.
- **Работа с данными**: запросы к БД через SQLAlchemy (чтение/создание/обновление/удаление).
- **Связь с фронтом**: передача данных в шаблоны (Jinja2) и редиректы.

**Файлы, за которые отвечаешь:** в основном `app.py` (вся логика). Модели `Doctor`, `Slot`, `Appointment` и `BookingForm` — совместно с Database Engineer (он проектирует ERD и структуру таблиц; ты используешь модели в маршрутах).

## 3. Маршруты (API / endpoints)

### 3.1 `GET /` — главная страница

**Что делает:**

1. Вычисляет текущее время и горизонт 7 дней вперёд.
2. Загружает всех врачей и все слоты в этом диапазоне.
3. Собирает список уникальных дат, на которые есть слоты (`available_dates`).
4. Читает из URL параметр `?day=YYYY-MM-DD`. Если он передан и дата есть в `available_dates`, выбирается эта дата; иначе — первая доступная.
5. Фильтрует слоты только по выбранной дате и группирует по `doctor_id` в словарь `slots_by_doctor`.
6. Передаёт в шаблон: `doctors`, `slots_by_doctor`, `available_dates`, `selected_date`.

**Зачем:** чтобы на одной странице показывался один день — без «огромного списка» всех дней. Фронт рисует кнопки дней и под каждой карточкой врача — только слоты этого дня.

---

### 3.2 `GET /slots/<slot_id>/book` — форма бронирования

**Что делает:**

1. Находит слот по `slot_id` (`Slot.query.get_or_404(slot_id)`).
2. Если слот уже занят (`not slot.is_free`), показывает сообщение и редирект на главную.
3. Если метод GET — отдаёт шаблон `book.html` с объектом `slot` (врач, дата, время).

**Зачем:** пользователь видит, какой слот выбрал, и форму для имени и телефона.

---

### 3.3 `POST /slots/<slot_id>/book` — создание записи

**Что делает:**

1. Снова загружает слот и проверяет `slot.is_free` (на случай если кто-то уже забронировал пока заполняли форму).
2. Читает форму через `BookingForm.from_request()` и валидирует (`validate()`). Если есть ошибки — возвращает `book.html` с кодом 400 и сообщениями в `flash`.
3. **Логика «один слот — одна запись»:**
   - Если у слота уже есть связанная запись `Appointment` и её статус `BOOKED` — редирект с ошибкой.
   - Если запись есть, но статус отменён (или мы удалили её логически) — **переиспользуем эту строку**: обновляем `patient_name`, `phone`, `status="BOOKED"`, новый `cancel_token`, `created_at`. Так мы не нарушаем ограничение БД `slot_id UNIQUE`.
   - Если записи нет — создаём новую `Appointment` и добавляем в сессию.
4. `db.session.commit()`. При любой ошибке (например, уникальность) — rollback, сообщение «slot may have been taken», редирект на главную.
5. При успехе — рендер `success.html` с объектом `appointment` (чтобы показать данные визита и ссылку на отмену).

**Зачем:** гарантировать, что один и тот же слот нельзя забронировать дважды и что после отмены слот можно снова забронировать без ошибки БД.

---

### 3.4 `GET /cancel/<token>` — отмена по ссылке

**Что делает:**

1. Ищет запись по `cancel_token`: `Appointment.query.filter_by(cancel_token=token).first()`.
2. Если не найдена — `abort(404)`.
3. Удаляет запись из БД: `db.session.delete(appt)` и `db.session.commit()`.
4. Сообщение «Appointment canceled. The slot is now free again.» и редирект на главную.

**Зачем:** пациент может освободить слот по ссылке из письма/страницы подтверждения; слот снова становится свободным и отображается с кнопкой «Book».

---

### 3.5 `GET /admin/appointments` — список всех записей

**Что делает:**

1. Выполняет запрос с джойнами: `Appointment` → `Slot` → `Doctor`, сортировка по `Slot.start_at` по убыванию.
2. Передаёт список `appointments` в шаблон `admin.html`.

**Зачем:** администратор видит все бронирования в одной таблице (врач, время, пациент, телефон, статус).

## 4. Вспомогательные элементы (твоя зона)

- **`create_app()`**  
  Ты настраиваешь приложение: `SECRET_KEY`, `SQLALCHEMY_DATABASE_URI` (SQLite по умолчанию), вызов `register_routes(app)` и `register_cli(app)`. Паттерн **Factory**: одна функция создаёт и возвращает настроенное приложение.

- **`BookingForm`**  
  Ты читаешь данные из `request.form` и валидируешь (имя не короче 2 символов, телефон не короче 5). Это отдельный слой между HTTP и моделями — удобно тестировать и менять правила без трогания маршрутов.

- **Обработка ошибок**  
  В `book()` при исключении после `commit()` ты делаешь `rollback()` и редирект с сообщением — пользователь не видит 500, а получает понятное «Try another slot».

---

## 5. Design patterns (что сказать на защите)

- **Factory:** `create_app()` — фабрика приложения Flask.
- **Singleton (по смыслу):** один экземпляр `db = SQLAlchemy()` на всё приложение.
- **Facade:** маршрут `book()` скрывает за собой валидацию, проверку слота, создание/обновление записи и работу с сессией БД.
- **Decorator:** использование декораторов Flask `@app.get`, `@app.route` для привязки URL к функциям.

«Я отвечаю за backend: настройку приложения и всю бизнес-логику в одном файле `app.py`.

Приложение создаётся функцией `create_app()`: там задаётся конфигурация, подключается SQLite через Flask-SQLAlchemy и регистрируются маршруты. Это паттерн Factory.

Главная страница — маршрут `index`. Я читаю из URL параметр `day`, выбираю одну дату из доступных и отдаю в шаблон только слоты этой даты, сгруппированные по врачу. Так на экране всегда один день, без длинного списка.

Бронирование — маршрут `book` для GET и POST. При GET я показываю форму; при POST читаю имя и телефон через класс `BookingForm`, валидирую их и создаю или обновляю запись в таблице `Appointment`. В базе у слота может быть только одна активная запись — это ограничение по `slot_id`. Если пользователь раньше отменил запись, мы переиспользуем ту же строку, чтобы не нарушать уникальность. После успешного сохранения показывается страница подтверждения с ссылкой на отмену.

Отмена реализована в маршруте `cancel`: по токену из URL я нахожу запись и удаляю её из базы. Слот снова становится свободным.

Админ-страница — маршрут `admin_appointments`: один запрос с джойнами по слотам и врачам, сортировка по времени, и все данные передаются в шаблон.

Ошибки обрабатываю через try/except при commit: при сбое делаю rollback и редирект с сообщением, чтобы пользователь не видел техническую ошибку.»
